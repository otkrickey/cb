# Documentation レビューガイドライン

## 対象

`docs/` 配下および `CLAUDE.md` の変更（設計書, ステータス, 実装計画, レビューガイドライン）

## レビュー方針

ドキュメントレビューの最優先事項は**ドキュメント間の内容レベルの整合性**である。リンクの存在やフォーマットの確認だけでなく、参照先の内容を実際に読み、記述が矛盾していないかを検証すること。

## 評価項目

### 1. ドキュメント間の内容整合性（最重要）

変更されたドキュメントの種別に応じて、以下の整合性を**参照先の内容を読んで**確認する。

#### 設計書（design/modules/, design/flows/）が変更された場合

- **実装計画との整合**: 設計書の公開API・データ構造が `docs/resolved/` の実装タスクと一致するか
- **フロー図との整合**: `docs/design/flows/` のモジュール間呼び出し関係が設計書の公開APIと一致するか
- **コードとの整合**: 設計書の型名・関数シグネチャが実際のコードと一致するか

#### ステータス（status/）が変更された場合

- **コードとの整合**: `implementation.md` のステータスが実際のコードベースと一致するか
- **設計書との整合**: `roadmap.md` の将来計画が設計書のスコープ境界と矛盾しないか
- **陳腐化の検出**: 既に実装済みの機能が `not-started` のままになっていないか

#### 実装計画（plans/, resolved/）が変更された場合

- **設計書との整合**: 計画の実装対象が設計書の仕様に基づいているか
- **他の実装計画との整合**: 計画間の依存関係が正しいか、重複がないか

### 2. コードベースとの整合

- 設計書・実装計画に記載された**型名・関数名・ファイルパス**がコードベースに存在するか
- FFI関数の数（現在11関数）やテスト数（現在23テスト）が正しいか

### 3. 用語の一貫性

- CLAUDE.md の用語や技術スタック記述（SQLCipher, swift-bridge, Carbon Event Manager等）と各ドキュメントの表記が統一されているか

### 4. 形式面（補助的）

- 各サブディレクトリの GUIDE.md の規約に従っているか
- ADR間の相互参照が正しいか
